mini-memory Hindsight Provider 实现与 Hermes Agent 对比

本报告说明当前 src/minisweagent/memory/providers/hindsight.py 的实现方式， 它如何接入 MemoryAgent / MemoryManager，以及它相对 hermes-agent/plugins/memory/hindsight 的取舍、已补齐能力和仍然缺失的能力。

代码状态：基于当前工作树中已实现的 Hindsight 修复，包括任务结束同步写入、外部 PostgreSQL、initial recall 和 tags/filter。

总体结论 当前实现 生命周期 配置 Hermes 对照 能力对比 SWE-bench-Pro 影响 剩余差距

1. 总体结论

当前定位

mini-memory 的 Hindsight provider 是一个面向 SWE-bench 类代码任务的轻量本地 provider。 它只支持 Hindsight local_embedded 路径，通过 embedded daemon + PostgreSQL/pgvector 实现长期记忆抽取、召回和 reflect。

近期已补齐

已补齐几个会直接影响实验可信度的问题：同一 document_id 下使用 update_mode="append"，但自动 transcript 不再每个 tool step 后写入； sync_turn 只 buffer，on_session_end 用 retain_async=false 同步写一次完整 task transcript，并支持外部 PostgreSQL 与初始 recall overlay。

与 Hermes 的主要差异

Hermes 是完整交互式 agent 产品的 provider：支持 cloud/local_external/local_embedded、 per-turn auto prefetch、多 profile/gateway metadata、session switch、配置向导和更多控制旋钮。 mini-memory 保留更小的接口面，重点服务链式实验。

最关键的实验含义：现在 mini-memory Hindsight 已经可以作为 SWE-bench-Pro 的一个消融变量运行， 但它不是 Hermes 的完整 hybrid memory 行为。它默认只有一次初始自动召回，之后仍主要依赖模型主动调用 hindsight_recall / hindsight_reflect。

2. 当前 mini-memory Hindsight provider 怎么实现

2.1 代码边界

文件 职责
src/minisweagent/memory/provider.py 定义最小 MemoryProvider ABC：tool schemas、tool dispatch、静态 prompt、initial context、turn sync、session end、builtin mirror 和 shutdown。
src/minisweagent/memory/manager.py 组合内置 MEMORY.md、可选外部 provider、SQLite session search 和 consolidation；负责工具路由和生命周期分发。
src/minisweagent/agents/memory.py 把 memory/provider tools 注入模型 extra_tools，路由 memory tool call，并在 run/step/finally 中触发生命周期。
src/minisweagent/memory/providers/hindsight.py 封装 Hindsight embedded daemon，暴露 hindsight_retain、hindsight_recall、hindsight_reflect 三个工具。

2.2 Provider 的内部结构

同步写入层

sync_turn() 把每个已完成 turn 序列化为 JSON 并保存在内存 buffer。 Hindsight 不再在每轮 tool call 后写后端；任务结束时由 on_session_end() 同步调用一次 aretain_batch()，写完后 runner 才进入下一个 issue。

异步 client 层

provider 内部维护一个后台 asyncio loop。Hindsight client 的 async 方法通过 asyncio.run_coroutine_threadsafe() 派发到这个 loop，再用 request_timeout 控制等待时间。

2.3 三个工具

工具 用途 当前参数 返回
hindsight_retain 主动写入一段叙事型工程经验，让 Hindsight 抽取实体、事实和索引。 content 必填，context 可选，tags 可选。 {"success": true, "message": "Memory stored."} 或结构化错误。
hindsight_recall 按 query 搜索长期记忆，返回 ranked memory text。 query 必填；provider 配置可附加 recall_tags 和 tags_match。 {"success": true, "results": [...], "count": n}。
hindsight_reflect 对已有长期记忆做 LLM synthesis，适合问“类似问题以前有什么经验”。 query 必填。 {"success": true, "answer": "..."}。

2.4 关键可靠性机制

机制 当前实现 为什么重要
append 写入 _retain_session_transcript() 的单个 task transcript item 带 update_mode="append"。 保留 append 语义，同时避免每个 tool step 都触发 Hindsight extraction。
失败传播 任务结束写入是同步调用；retain 失败会从 on_session_end() 抛出，不再被后台 writer 静默吞掉。 正式链式实验能更早暴露 Hindsight 后端不可用或 extraction 失败。
同步任务边界 on_session_end() 强制 retain_async=false；shutdown() 只负责关闭 client 和 loop。 链式实验中一个 instance 结束时，经验已写入后才会服务下一个 instance。
stale daemon retry 遇到 connection refused、server disconnected 等连接类错误时，清空 client 并重试一次。 embedded daemon idle shutdown 或短暂断连时，不让一次工具调用直接失败。
external Postgres database_url 透传给 HindsightEmbedded，最终进入 daemon env。 可以使用外部带 pgvector 的 PostgreSQL，绕过 pg0/pgvector 二进制和宿主 glibc 不兼容问题。
fail-fast fail_fast=True 时 initialize 会调用 client warmup。 用于实验启动时提前暴露 Hindsight 环境问题，避免跑完才发现 memory backend 不健康。

3. 生命周期与数据流

1. Agent 构造 MemoryAgent.__init__ 从配置构造 MemoryManager，把 memory/Hindsight tools 追加到模型 extra_tools。
2. run 初始化 manager.initialize(session_id) 加载 frozen MEMORY snapshot，并初始化 Hindsight profile/bank/document。
3. 初始召回 initial_context(task) 可用 task 做一次 Hindsight recall，并把结果 prepend 到首个 user message。
4. 工具调用 模型可调用 memory、session_search、hindsight_*。provider tool 由 manager 路由。
5. turn sync 每轮工具 observation 追加前，agent 把 previous user/tool + assistant summary 交给 provider；Hindsight 只写入内存 buffer。
6. finally 收尾 on_session_end() 同步 retain Hindsight transcript、记录 session store；owned manager 再 shutdown。

3.1 初始化细节

HindsightProvider.initialize() 会基于传入 session_id 创建 document_id = session_id + UTC timestamp。默认 profile 和 bank_id 为 mini-memory 时，会用 memory.home 的 sha1 hash 派生 scoped name，避免不同 chain 的 memory home 共用同一 Hindsight bank。

3.2 初始召回 overlay

mini-memory 没有实现 Hermes 的 per-turn prefetch，而是新增了一个更小的 initial_context(query) 钩子。它只在每个 run 开始时用原始 task 做一次 recall， 将结果包装为 <memory-context> 块并 prepend 到首个 user message。 这样不会修改 system prompt，也不破坏 frozen MEMORY snapshot 的语义。

3.3 写入路径

sync_turn(user, assistant)
  -> append JSON turn to _session_turns
  -> no Hindsight network write on the tool-call hot path

on_session_end(messages)
  -> build one full task transcript from _session_turns
  -> aretain_batch(bank_id, items=[... update_mode="append" ...], document_id=..., retain_async=False)
  -> on success: clear _session_turns
  -> on failure: propagate the retain error

3.4 内置 MEMORY.md 镜像

MemoryManager._handle_builtin() 在内置 memory 工具写入成功后调用 provider.on_memory_write()。Hindsight provider 会同步镜像 add 和 replace，跳过 remove 和空内容。这样紧凑 curated facts 可以进入 Hindsight 长期记忆，但当前 consolidation 直接写 BuiltinMemory 的路径仍不经过 manager tool，因此不会自动镜像。

5. Hermes Agent Hindsight provider 怎么实现

Hermes 的 Hindsight provider 是面向完整 Hermes 产品的 plugin，实现范围明显更大。它不仅是 local embedded adapter，也负责配置加载、setup wizard、cloud/local_external 模式、gateway/session/profile metadata、auto recall/prefetch、session switch 和 compression 生命周期。

5.1 Hermes 的核心能力

能力 Hermes 实现方式
多模式 支持 cloud、local_embedded、local_external；不同模式安装不同依赖。
配置来源 从 profile-scoped config、legacy config 和环境变量读取；还有 setup wizard 保存配置和 secret。
Auto recall queue_prefetch() 在上一 turn 后后台 recall；prefetch() 在下一 turn 前取出结果并注入上下文。
Memory mode 支持 hybrid、context、tools：可以只注入上下文、只提供工具，或两者兼有。
Retain 节流 支持 auto_retain 和 retain_every_n_turns，可以每 N turn 写一次。
Append 能力探测 通过 /version 探测 Hindsight API 是否支持 update_mode="append"，支持则用 session-scoped document append，否则回退到 per-process document。
Tags/filter 支持 default retain tags、per-call tags、recall tags、tags_match、recall types。
Bank 模板 支持 bank_id_template，可按 profile/workspace/platform/user/session 派生 bank。
Session switch 实现 on_session_switch()，在 resume/branch/reset/new/compression 时 flush 旧 buffer、清 prefetch、记录 parent lineage。
Daemon 管理 local embedded 初始化时后台启动 daemon，写日志；配置变化时重启 daemon；连接断开时重建 client 重试。

5.2 Hermes 的 prefetch 机制

Hermes 的 auto recall 是两阶段的：上一轮结束时 queue_prefetch(query) 后台运行， 下一轮开始前 prefetch() 取出缓存结果并注入当前 user context。它不会把 recall 结果写进 system prompt，而是作为非缓存层的临时上下文。这对交互式 agent 很有用，因为模型不需要主动记得调用 hindsight_recall。

5.3 Hermes 的 session switch 机制

Hermes 是长期运行的多会话 agent，session id 会在 resume、branch、reset、new、compression 等路径中变化。 Hindsight provider 会在切换前 flush 旧 session 的 buffer，切换后清空 turn buffer 和 prefetch result， 并把 parent/session lineage 放入 tags/metadata。这对网关、多用户和长会话很重要。

6. mini-memory 与 Hermes 的逐项对比

维度 mini-memory 当前实现 Hermes Agent 实现 评价
目标场景 SWE-bench 类自动代码任务，链式 trial 间复用工程经验。 通用交互式 agent，覆盖 CLI、gateway、多 profile、长期用户会话。 取舍合理 mini-memory 更窄、更实验导向。
模式支持 只支持 local_embedded，但可通过 database_url 使用外部 PostgreSQL。 支持 cloud、local_embedded、local_external。 能力较少 如果要服务大规模并行，local_external HTTP server 更好管理。
模型配置 默认 ModelHub gpt-5.4-2026-03-05，走 chat-completions endpoint。 按 provider 映射默认模型，如 OpenAI 默认 gpt-4o-mini。 贴合主实验 但要记住 Hindsight 不走 Responses API。
Tool schema 返回 mini-swe-agent 使用的 OpenAI wrapper 格式：{"type":"function","function":...}。 Hermes plugin 返回 unwrapped tool schema。 适配本项目 与 model action parser 对齐。
Prompt 注入 静态 provider block 进入 system prompt；initial recall 进入首个 user message。 静态 block + per-turn prefetch context overlay。 更保守 cache 友好，但自动召回频率低于 Hermes。
自动召回 仅 auto_recall_on_init 一次，之后靠工具主动调用。 可每 turn 后台 prefetch，下 turn 注入。 可能低估 Hindsight 如果模型不主动 recall，收益会变小。
自动写入 每 turn 后只 buffer；任务结束时同步写一次完整 task transcript。 支持 auto_retain 和 retain_every_n_turns。 更适合 SWE-bench 避免每个 tool step 触发 extraction，同时保留完整任务上下文。
Append 语义 直接使用 update_mode="append"。 先探测 API 版本，支持 append 则用，否则回退。 当前可用 依赖 Hindsight client/API 版本较新；可后续补 version probe。
失败保留 任务结束同步 retain；失败会直接传播，不再后台重试。 writer 捕获异常并记录；更完整地处理 session switch flush。 偏 fail-fast 更利于实验发现后端问题，但没有后台自动重试。
Shutdown 自动 transcript 已在 on_session_end 同步完成；shutdown 关闭 client 和本 provider 的 asyncio loop。 writer join 10s，prefetch join，特殊处理 embedded inner client，不停全局共享 loop。 不同并发模型 mini 每 provider 一个 loop；Hermes 用 module-global loop 支持多 provider 实例。
Tags/filter 支持 retain_tags、tool tags、recall_tags、recall_tags_match。 支持 retain tags、recall tags、tags_match、types、source、lineage tags。 基础已补齐 但 lineage 和 types 还没有。
Bank 隔离 默认按 memory.home hash 派生 profile/bank；显式配置则完全尊重用户指定。 支持 bank_id_template，可用 profile/workspace/platform/user/session。 适合 chain home 隔离 但不如模板灵活。
Session switch 无 on_session_switch。 完整实现，处理 resume/branch/reset/new/compression。 当前非主要问题 mini-swe-agent 每 instance run 边界清晰；未来 resume/compression 要补。
Setup/依赖管理 通过 optional dependency [hindsight] 安装；无 setup wizard。 setup wizard 按模式安装依赖、保存 config/env。 实验环境需脚本化 mini 更适合 config overlay + runner。

7. 对 SWE-bench-Pro 任务性能的影响

7.1 已经解决的高风险点

问题 修复后状态 对实验的意义
同一 document 下 turn 覆盖 已修 任务结束单次 transcript item 仍带 update_mode="append"，避免同 document replace 语义误覆盖。 完整任务轨迹作为一个上下文更完整的 document 写入。
每个 tool step 后写入过慢且噪声大 已改 sync_turn 只 buffer，Hindsight 不在 tool-call 热路径上写后端。 减少 LLM extraction 次数和 daemon 压力，召回内容拥有更完整任务上下文。
实例结束时记忆尚未可见 已改 on_session_end 强制 retain_async=false 同步写入。 step N 的经验写完后才进入 step N+1。
backend 不健康但实验继续 可配置 fail_fast 可在 initialize 时提前启动/检查 daemon。 正式实验建议打开 fail-fast，避免把基础设施失败误判为 memory 方法无效。

7.2 仍会影响 Hindsight 上限的取舍

mini-memory 当前不是 Hermes hybrid 模式。它只在实例开头做一次自动 recall，后续不做 per-turn prefetch。 如果模型在解题中不主动调用 hindsight_recall，Hindsight 的收益会低于 Hermes 自动注入上下文的上限。
取舍 可能影响 建议
只做 initial recall 任务中途出现的新线索不会自动触发更精准 recall。 先保留；如果 Hindsight 消融收益偏低，再加 per-turn 或 every-N-turn prefetch overlay。
无 retain_every_n_turns 已不再每 turn 写 Hindsight；但 task transcript 仍可能很长。 后续可增加 max chars、结构化摘要或 session-end summarizing retain。
无 repo/chain 自动 tags 当前 tags 需要 overlay 显式配置，否则同 bank 内召回可能混杂。 chain runner 可在 overlay 中注入 retain_tags/recall_tags；后续可自动从 instance metadata 生成。
task-end retain 失败会中断链 这是刻意的 fail-fast 行为：下个 issue 不能在记忆未写完时开始。 正式实验应保证外部 PostgreSQL/daemon 稳定，并适当增大 request_timeout。

8. 仍然缺失或值得后续补的能力

优先级 能力 当前状态 建议
P1 API version probe mini-memory 直接使用 update_mode="append"，没有像 Hermes 那样探测 API 版本。 如果实验环境版本固定且新，风险低；若要跨机器复现，应补 /version probe。
P1 正式 fail-fast overlay 代码已有 fail_fast，但默认示例仍是 false。 正式 Hindsight 实验 overlay 建议设为 true，避免 silent degradation。
P2 自动 repo/chain tags 有通用 tags 配置，但还不会自动从 SWE-bench instance/chain metadata 派生。 在 benchmark runner 层将 repo、chain_id、instance_id 注入 provider config 或 initialize kwargs。
P2 task transcript 清洗/摘要 当前任务结束写入完整 buffered turns，可能包含长日志和无效探索。 增加 max chars、跳过超大日志，或在 session end 先生成结构化摘要再 retain。
P2 per-turn prefetch mini-memory 当前只做 initial recall。 如果想评估 Hindsight 上限，可增加 cache-friendly user overlay，不要改 system prompt。
P3 local_external HTTP 模式 当前只有 embedded client + 可选外部 database。 大量并行 worker 时，独立 Hindsight server 可能更好运维；可后续补。
P3 session switch/compression hooks mini-swe-agent 当前 run 边界明确，未实现。 如果未来引入 resume/compression/长驻 agent，需要补齐。
