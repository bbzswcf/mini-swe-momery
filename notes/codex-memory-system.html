Codex Memory 文件记忆系统解析

结论先说清楚：Codex 的 memory 确实是一个文件化记忆系统，但运行时模型通常读到的不是历史 rollout 全貌， 而是后台从历史 rollout 抽取、筛选、合并后的多层文件记忆。完整 rollout 是写入管线的原始证据，不是每次任务默认加载给模型的上下文。

一句话模型

Codex 把历史对话/工具调用的 rollout 先抽成结构化 raw_memory， 再由一个内部 consolidation agent 合并成 ~/.codex/memories/ 下的文件： memory_summary.md、MEMORY.md、rollout_summaries/， 以及可选的 skills/。新任务开始时，模型只被提示“有这些记忆文件可查”，并直接收到一个很短的 memory_summary.md 摘要。

模型能看到什么，不能看到什么

运行时默认可见

当 memory feature 和 use_memories 都开启，并且 ~/.codex/memories/memory_summary.md 存在且非空时，Codex 会把一段 “Memory” developer policy 注入 prompt。这段 policy 包含：

  • memory_summary.md 的截断内容，约 2500 token 上限；
  • memory 文件夹布局说明；
  • 什么时候应该查 memory 的决策边界；
  • 引用 memory 的隐藏 citation 格式要求。

运行时默认不可见

过去完整 rollout 不会直接塞进新任务上下文。模型也不会自动看到所有 rollout_summaries/*.md、完整 MEMORY.md 或 原始 .jsonl 历史。它需要按 prompt 指引主动搜索/读取相关记忆文件。

换句话说，Codex 的 memory 是 retrieval-oriented 的，而不是“把历史聊天全量续上下文”。

文件夹结构

  • ~/.codex/memories/：memory 根目录，也是 Phase 2 consolidation agent 的工作区。
  • memory_summary.md：最小 prompt-loaded 层，必须以 v1 开头，目标是短、密、高路由价值。
  • MEMORY.md：主 handbook，按 task group 聚合，便于 grep 和人工/模型扫描。
  • raw_memories.md：Phase 2 输入层，由 SQLite 中选中的 stage1 outputs 机械合并而来。
  • rollout_summaries/：每个被选中 rollout 的摘要文件，保存比 MEMORY.md 更接近证据的细节。
  • skills/<skill-name>/：可选的可复用流程文件，entrypoint 是 SKILL.md。
  • extensions/：显式用户更新或外部 memory extension 的输入区。
  • .git/：Codex 在 memory root 下维护 git baseline，用 diff 判断 Phase 2 是否有实际变化。

层次关系

层 作用 运行时是否直接进 prompt 信息密度
memory_summary.md 全局短摘要 + 路由索引 + 用户偏好/通用提示。 是，会被注入 developer policy。 最高压缩。
MEMORY.md 主记忆手册，按任务族保存可复用经验、失败规避、命令和路径。 否，prompt 告诉模型需要时搜索/读取。 中等压缩。
rollout_summaries/ 单个 rollout 的摘要和证据片段，用于需要更精确上下文时下钻。 否，只有被 MEMORY.md 指向时才建议打开。 较少压缩。
原始 rollout 历史对话、工具调用、工具输出的原始记录。 否；Phase 2 prompt 明确要求不要打开 raw sessions。 原始证据。

写入机制：从 rollout 到文件记忆

Codex 的写入不是“模型在完成任务时自己写 MEMORY.md”。它是后台 pipeline，在 root session 有输入后异步启动。 入口会跳过 ephemeral session、feature 未开启、sub-agent session、state DB 不可用等场景。

1. 历史 rollout 进入 state DB 交互线程有 rollout path、cwd、updated_at、source、memory_mode 等元数据。
2. Phase 1 claim 后台扫描最近、空闲足够久、memory_mode=enabled、尚未抽取或已过期的线程。
3. Phase 1 extraction 模型读取过滤后的 rollout items，输出 JSON：raw_memory、rollout_summary、rollout_slug。
4. Phase 2 selection 按 usage_count、last_usage、source_updated_at 等选择要参与全局合并的 raw memories。
5. Consolidation agent 内部 agent 在 memory root 内更新 MEMORY.md、memory_summary.md、skills/。

Phase 1：单 rollout 抽取

Phase 1 从 state DB claim 一批合格历史线程，读取 rollout 文件，只保留 memory-relevant 的 response items。 它会过滤掉 developer messages，并剔除 AGENTS.md 注入块和 skill 注入块，避免把系统/项目指令错误记成用户经验。 

stage1 output schema:
{
  "raw_memory": "... detailed markdown raw memory ...",
  "rollout_summary": "... compact routing summary ...",
  "rollout_slug": "... optional stable-ish filename slug ..."
}

Phase 1 的输出写入 SQLite stage1_outputs，不是直接写文件。这里的 raw_memory 是“从完整历史中抽出来的结构化记忆原料”，它比完整 rollout 小得多，也更适合后续合并。

Phase 2：全局文件合并

Phase 2 先拿全局 lock，然后把当前选中的 stage1 outputs 同步成文件输入： raw_memories.md 和 rollout_summaries/*.md。 Memory root 是一个 git baseline 目录，Codex 会生成 phase2_workspace_diff.md， 再启动一个内部 consolidation agent 读取 diff 和输入文件，修改最终记忆文件。

这个内部 agent 被严格限制：cwd 是 memory root，ephemeral，memory feature 关闭，不允许递归生成 memory； 无网络，approval policy 是 never，只有 memory root 写权限。

读机制：新任务如何使用 memory

读路径由 extension 贡献 developer policy。它读取 memory_summary.md，渲染 read_path.md 模板，并把结果作为 developer policy 注入。这个模板告诉模型：

  • 先看已注入的 summary，提取关键词；
  • 用关键词查 MEMORY.md；
  • 只有 MEMORY.md 明确指向时，再打开 1-2 个最相关的 rollout summary 或 skill；
  • 保持 quick pass，通常 4-6 步内停止；
  • 如果用到了 memory，最终回复尾部追加隐藏 <oai-mem-citation>。

当前工具状态

Codex 仓库里有两套 read-only memory 工具实现：一个 extension tool 版本，一个 MCP server 版本， 都提供 list、read、search。 但 app-server 当前安装 extension 时只启用了 prompt contributor，tool contributor 被注释掉。 所以当前主路径更像“把 memory 文件夹和检索规则告诉模型”，而不是强制走专门 memory tool。

SQLite 与 usage 反馈

文件不是唯一状态。Codex 用 SQLite 管后台 pipeline 的可靠性和 selection：

状态 用途
stage1_outputs 每个 thread 的 raw_memory、rollout_summary、rollout_slug、生成时间、usage 统计、Phase 2 选择标记。
jobs Phase 1 和 Phase 2 的 job lease、retry backoff、watermark、ownership token。
threads.memory_mode enabled、disabled、polluted 等模式，控制某条历史是否可被抽成 memory。
usage_count / last_usage 当模型在最终回答中引用 memory citation，Codex 会更新相应 stage1 output 的使用统计，后续 Phase 2 selection 会优先保留被用到的记忆。

安全与遗忘机制

避免污染

如果配置 disable_on_external_context 开启，使用 web search / external context 的线程会被标记为 polluted，不再作为 memory 输入。

避免 prompt injection

Phase 1 明确把 rollout 内容视为数据而不是指令，并过滤 developer/AGENTS/skill 注入片段。

遗忘靠 selection + diff

Phase 2 当前选中集合会直接同步到文件夹；不再选中的 rollout summary 会删除。Consolidation agent 通过 git diff 做对应清理。

和 AGENTS.md 的关系

AGENTS.md 是项目/目录指令，不是 Codex memory pipeline 的输出。Codex 会从项目根到当前目录收集 AGENTS.md 或 AGENTS.override.md，作为 user instructions 注入。 Phase 1 反而会过滤这类注入块，避免把“本次项目指令”误当成历史经验。

对 mini-memory 的参考价值

对 SWE-bench 类任务，Codex 的核心思想很有价值：把历史 trial 全貌留作证据，但运行时只暴露逐层压缩后的文件记忆。 不过 Codex 的默认 prompt 强调 user profile、communication preferences、skills，这和 mini-memory 的非目标不同。

更适合 mini-memory 的改造是：

  • memory_summary.md：repo / package / issue-family 的短路由索引；
  • MEMORY.md：patch patterns、失败规避、测试命令、环境坑、可复用定位路径；
  • rollout_summaries/：保存 trial 级证据和验证结果；
  • 去掉 user profile 和面向通用个人助手的偏好记忆；
  • 保留 Phase 1/Phase 2 解耦和 citation/usage 反哺 selection。

源码锚点

  • codex-rs/memories/README.md：官方 memory pipeline 概览。
  • codex-rs/memories/read/src/prompts.rs：读取 memory_summary.md 并构造 developer instructions。
  • codex-rs/memories/read/templates/memories/read_path.md：运行时模型看到的 memory 使用规则。
  • codex-rs/memories/write/src/start.rs：后台 memory startup task 入口。
  • codex-rs/memories/write/src/phase1.rs：从 rollout 抽取 raw_memory。
  • codex-rs/memories/write/src/phase2.rs：文件同步、diff、consolidation agent。
  • codex-rs/memories/write/templates/memories/consolidation.md：MEMORY.md、memory_summary.md、skills/ 的格式要求。
  • codex-rs/state/src/runtime/memories.rs：SQLite job claim、selection、usage、pollution。
  • codex-rs/ext/memories/src/extension.rs：memory prompt/tool extension 注册点。
  • codex-rs/memories/mcp/src/server.rs：read-only memory MCP server。
  • codex-rs/core/src/agents_md.rs：AGENTS.md discovery 和注入。
